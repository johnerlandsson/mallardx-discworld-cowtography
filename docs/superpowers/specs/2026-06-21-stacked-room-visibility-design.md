# Stacked Room Visibility Design

## Problem

Many maps have multiple rooms at the same x/y coordinate (739 stacked positions in the current dataset, some with up to 12 rooms). These are multi-floor buildings where each floor occupies the same map cell. Currently all floors are drawn on top of each other, producing a cluttered, meaningless symbol.

## Goal

Show only the ground floor room by default. When the player enters any room in a stacked group, show only that room (and its stair symbol). Restore ground floor defaults when the player leaves the stack.

---

## Architecture

### Build time

A new `buildStackData(rooms, exitPairs, stairRooms, overrides)` function in `build-svg.mjs` detects stacking groups and identifies the ground floor room for each. It outputs `ui/data/room-stacks.js`, which `mapper.js` imports at runtime.

### Runtime

`mapper.js` applies initial stack state on every map load (hiding upper-floor rooms), tracks which stack the player is currently inside, and swaps visibility on position change.

### Override

`ui/data/room-ground.json` lets authors pin the ground floor for any position where the heuristic fails. Starts empty.

---

## Section 1: Build-time stacking detection

### Inputs

- Full rooms table from DB: `{ id, mapId, x, y }`
- Exit pairs with `isVertical` flag (already computed by existing `buildExitPairs`)
- `stairRooms`: `Map<roomId, { hasUp, hasDown }>` (already computed)
- `ui/data/room-ground.json`: `{ "mapId:x:y": groundRoomId, ... }`

### Algorithm

**Step 1 — Group by position.**
Build `Map<"mapId:x:y", roomId[]>`. Positions with more than one room are stacks.

**Step 2 — Build non-vertical adjacency map (per map).**
From `exitPairs` where `isVertical=false`, build `Map<roomId, Set<roomId>>` covering all rooms on the map. Used for BFS in Step 3.

**Step 3 — BFS reachability score.**
For each room in a stacked group, do a BFS up to depth 5 following non-vertical exits, excluding rooms that are members of the same stacked group. The score is the count of unique rooms reached.

**Step 4 — Select ground floor.**
Ground floor = room with the highest BFS score.
- Tie-break 1: prefer the room where `stairRooms` has `hasDown=false` (up-only stair — not a basement).
- Tie-break 2: first room in the group list.

**Step 5 — Apply overrides.**
If `room-ground.json` has an entry for this position (`"mapId:x:y"`), that roomId is used as ground floor unconditionally; Steps 3–4 are skipped for this group.

### Output: `ui/data/room-stacks.js`

```js
// Maps each non-ground room to its ground floor room.
export const upperToGround = {
  "upperId": "groundId",
  // ...
}

// Maps each ground floor room to its upper-floor siblings.
export const groundToUppers = {
  "groundId": ["upperId1", "upperId2"],
  // ...
}
```

Only non-ground rooms appear as keys in `upperToGround`. Only ground rooms that have uppers appear in `groundToUppers`. Unstacked rooms are absent from both.

This file is generated by `build-svg.mjs` alongside the SVG files (run via `npm run build:svg`). No separate build step.

## Section 2: Runtime JS visibility management

### Imports

`mapper.js` imports both maps:
```js
import { upperToGround, groundToUppers } from "./data/room-stacks.js";
```

### `setStackRoomVisible(svg, id, visible)`

Hides or shows three things for a given room:
1. `#room-{id}` — the room rect or circle
2. Its immediately-following sibling if it has class `room-type-label` (same sibling pattern used by the existing `_lift` mechanism)
3. `#stair-{id}` in `layer-stairs`

```js
function setStackRoomVisible(svg, id, visible) {
  const d = visible ? '' : 'none';
  const roomEl = svg.querySelector(`#room-${CSS.escape(id)}`);
  if (roomEl) {
    roomEl.style.display = d;
    const sib = roomEl.nextElementSibling;
    if (sib?.classList.contains('room-type-label')) sib.style.display = d;
  }
  const stairEl = svg.querySelector(`#stair-${CSS.escape(id)}`);
  if (stairEl) stairEl.style.display = d;
}
```

### Initial state on map load

After the SVG swaps in, hide all upper-floor rooms that belong to stacks present on this map:

```js
for (const [groundId, uppers] of Object.entries(groundToUppers)) {
  if (!currentSvg.querySelector(`#room-${CSS.escape(groundId)}`)) continue; // not on this map
  for (const upperId of uppers) setStackRoomVisible(currentSvg, upperId, false);
}
```

### Position tracking

A module-level variable tracks the ground room of the stack the player is currently inside (if any):

```js
let currentStackGround = null;
```

Reset to `null` on map change (same location where `target` is cleared today).

### `updateStackVisibility(newRoomId)`

Called just before `drawPositionIndicator` whenever position changes, and with `null` when the player leaves a known map (GMCP gap):

```js
function updateStackVisibility(newRoomId) {
  // Restore previous stacked group to defaults.
  if (currentStackGround) {
    setStackRoomVisible(currentSvg, currentStackGround, true);
    for (const u of groundToUppers[currentStackGround] ?? [])
      setStackRoomVisible(currentSvg, u, false);
    currentStackGround = null;
  }

  if (!newRoomId) return;

  // If entering an upper-floor room, swap visibility.
  const ground = upperToGround[newRoomId];
  if (ground) {
    setStackRoomVisible(currentSvg, ground, false);
    setStackRoomVisible(currentSvg, newRoomId, true);
    currentStackGround = ground;
  }
  // Ground floor rooms are already visible; no action needed.
}
```

### Interaction with position indicator

`updateStackVisibility` runs before `drawPositionIndicator`. This ensures the current room's element has `display:''` before `drawPositionIndicator` queries and lifts it into the overlay group.

## Section 3: Override mechanism

**File:** `ui/data/room-ground.json`

```json
{
  "1:127:942": "<roomId>"
}
```

- Key: `"{mapId}:{x}:{y}"`
- Value: the roomId that must be treated as ground floor for that position
- Read by `build-svg.mjs` before running BFS; skips BFS for any position that has an entry
- File starts empty (`{}`)
- Same authoring pattern as `room-compact.json`, `room-water.json`, etc.

## Out of scope

- Animating the transition between floors
- Showing a floor indicator label
- Handling the case where two *different* buildings happen to share the same map coordinate (rare; handle via `room-ground.json` override if it arises)
